將媒體交付到台式機，移動電視和聯網電視上的不同設備，需要多種不斷變化的技術:

  • 音頻和視頻編解碼器
  • 各種類型的視頻流包裝
  • 各種內容保護方案可防止內容被複製

在不使用動態交付的情況下，跨平台流傳輸通常需要多次存儲相同的內容，以支持向所有目標平台的交付。動態交付使Brightcove客戶可以將媒體交付到各種設備，而無需為每個受支持的設備和用例多次重新打包和存儲內容而產生的成本和運營開銷。動態交付可實時創建最佳的內容包，以交付和流式傳輸到所有受支持的設備和用例。隨著技術和受眾的發展，這還有助於確保交付架構的未來適應性。

借助Dynamic Delivery，客戶可以提供單個高質量（夾層）視頻文件。在提取過程中，我們會創建此文件的多個質量級別，以適合各種不同的屏幕尺寸和網絡條件。觀眾按下播放鍵時，Video Cloud使用有關請求設備的信息來確定預期的流技術的類型，例如，流打包的類型（HLS與DASH）（如果設備不支持任何HTTP，則為MP4）流），所需的DRM（例如Widevine，Fairplay，Playready）以及輔助內容，例如隱藏式字幕或替代音軌。然後，動態交付使用此信息為請求設備實時交付適當的內容包（即流的類型）。

架構概述

  1. 上載源文件時，轉碼過程將使用所選的攝取配置文件來確定要創建的再現集。
  2. 渲染存儲為一組連續的，碎片化的MP4文件，這些文件的各種比特率和分辨率與任何特定的交付格式無關。
  3. 當請求視頻時，動態交付過程將進行即時打包或JIT打包，這意味著它將根據指定要求將內容動態打包為適當的格式。這種包裝將考慮所需的特定傳送格式，所需的音頻和字幕軌道以及所需的任何加密/ DRM。
  4. 當播放設備使用Playback API返回的清單來請求播放媒體時，我們會對片段化的MP4文件進行及時打包，以提供適當的段格式，持續時間和加密。這些片段被緩存在CDN上，因此視頻傳遞仍然具有很高的性能。

動態交付過程被設計為完全託管的基於雲的服務。所有組件都在多個區域的雲基礎架構上運行，以確保高可用性，冗餘和可擴展性。

動態投放會產生哪些視頻格式？

動態交付會根據需要生成以下來源:

  • HLS（版本3、4和5）
  • 光滑的
  • 短跑
  • MP4（如果在攝取配置文件中指定了漸進式再現）

將創建以下DRM-packagedrenditions（取決於您的帳戶啟用了哪些內容保護）:

  • HLS +公平競賽
  • DASH + Widevine
  • DASH + Playready
  • 流暢+隨時可用

遷移過程

自2017年10月17日（星期二）起，默認情況下，所有創建的新Video Cloud帳戶均配置為使用動態交付。這包括試用帳戶，新發布者的帳戶以及現有發布者的新子帳戶。進行此更改後，所有新發布者和試用客戶都將自動受益於Dynamic Delivery的所有功能。

對於現有客戶，遷移過程分為兩個步驟。首先，將新的上傳內容轉換為利用動態交付（日向前遷移），其次，將現有內容從舊交付平台遷移到動態交付（向後目錄遷移）。

前一天遷移

轉向動態交付的第一步是過渡新的上傳內容以利用動態交付。為您的帳戶啟用動態投放是一項簡單的任務，可以由客戶支持來完成。為現有的Video Cloud帳戶啟用動態交付沒有任何危險，因為它不會修改任何現有的內容。只有明確攝取到Dynamic Delivery中的內容才會使用新系統。

從工作流程的角度來看，Video Cloud Studio中的幾件事情將有所不同。首先，使用“上傳"模塊上傳視頻時，您現在會在帳戶中看到“動態投放"攝取配置文件。

媒體模塊還將顯示一個綠色的雲圖標 () 在使用動態交付配置文件攝取的視頻標題旁邊。使用舊版攝取配置文件攝取的視頻將顯示一個灰色圖標 (）。

此外，由於以格式不可知的狀態存儲格式副本，因此媒體模塊中的“視頻源"面板看起來會略有不同。

從功能上講，其他所有內容都保持不變。您可以通過所有標準機制（Video Cloud Studio，Dynamic Ingest API，CMS連接器等）進行上傳，並且所有標準播放設備上的播放均保持不變。

使用Dynamic Delivery提取視頻後，如果對視頻進行了重新轉碼，則必須選擇Dynamic Delivery配置文件。如果您想使用舊的提取配置文件，則必須重新上傳視頻。

筆記: 動態傳遞引入了區域化攝取和傳遞的概念。在以前的傳送系統下，所有攝取處理和傳送源均位於美國。借助Dynamic Delivery，Brightcove可以配置一個帳戶，在該帳戶中進行提取，並且來源以逐個發布者為基礎，以提高性能。所有Video Cloud帳戶都將默認為“美國東部"，但可以重新配置為您的本地區域。請注意，轉移到另一個地區會涉及成本問題。如需進一步資訊，請聯絡您的 Brightcove 客戶經理。

返回目錄遷移

一旦您利用Dynamic Delivery獲得了新的摘要，我們便可以遷移您的後目錄。可以通過以下兩種方法之一來完成此操作:

Transmux遷移工具

Transmux遷移工具無需重新轉碼即可將Video Cloud內容從傳統交付轉移到動態交付。與重新轉碼內容相比，這使您可以更快，更輕鬆地遷移內容。因此，對於大多數客戶而言，它應該是首選的遷移方法。transmux工具將轉換您帳戶中的現有格式，使其與Dynamic Delivery兼容。要通過transmux工具遷移您的內容，請與您的客戶經理聯繫。

重新轉碼

有一些條件使標題不適合通過transmux工具進行遷移，因此需要重新編碼標題才能進行遷移。在以下情況下，標題將需要重新編碼:

  • 您希望利用上下文感知編碼
  • 您希望進行插播式服務器端廣告插入（SSAI）
  • 您的標題沒有HLS格式
  • 您的書名當前具有DRM（可以接受HLSe）

如果您需要重新轉換標題，則將需要自己（按照我們的最佳實踐）管理該過程，或者與您的客戶團隊合作以獲取付費的遷移服務。筆記: 可以對帳戶中的某些標題進行多路復用，而對其他標題進行重新轉碼。

經常問的問題

什麼是動態投放？

動態交付是使Brightcove的下一代媒體交付平台成為一組技術。動態交付旨在滿足發布商當前和未來的要求，並將成為所有Brightcove產品交付媒體的基礎。

動態投放有什麼好處？

  • 交付具有更大的靈活性和敏捷性-內容被動態打包以交付給許多端點（Web瀏覽器，智能手機，平板電腦，連接的電視和流媒體盒）。動態傳遞被設計為能夠使用多個CDN來傳遞內容，從而增加了客戶的選擇。
  • 面向未來-由於內容是動態打包的，因此Brightcove可以動態添加新格式。隨著新版本的HLS，DASH等，甚至是新格式的出現，我們只需在平台級別添加支持，發布者就可以開始使用它，而無需重新轉換現有內容。
  • 減少存儲空間-內容資產僅存儲一次，並且在需要時向下游創建表示形式以進行傳送，而不是多次存儲內容。
  • 通過地理分佈提高性能-動態交付專為區域分佈的高可用性體系結構而設計。發布者受益於改善的播放性能和增強的彈性。
  • 對服務器端廣告插入（SSAI）的集成支持-動態投放將使廣告或其他內容可以從服務器插入。這為觀看者提供了無縫的播放體驗，並使發布者增強了抵禦廣告攔截器的靈活性，而這一切都來自Video Cloud。

動態投放有什麼價格方面的考慮因素？

不，每個基本平台許可均包含對Dynamic Delivery的訪問。動態交付的某些功能（包括服務器端廣告插入（SSAI）和上下文感知編碼（CAE））是附加組件，具有相關的成本。請與您的客戶經理聯繫，以獲取有關價格的詳細信息。

動態交付是否支持BYO CDN發布者？

動態交付使發布者可以利用自己的CDN關係；但是，所有視頻資產必須由Brightcove Origin管理。請與您的客戶經理聯繫以獲取更多信息。

Dynamic Delivery支持在哪些區域進行攝取和來源？

Dynamic Delivery支持以下AWS區域中的提取和來源:

  • us-east-1（Virginia）-所有帳戶的默認設置
  • eu-west-1（都柏林）
  • ap-northeast-1（東京）
  • ap-southeast-1（新加坡）
  • ap-southeast-2（悉尼）

請注意，將攝取和來源轉移到另一個地區時會涉及成本問題。請聯繫您的Brightcove客戶經理以獲取更多信息。

所有Brightcove帳戶都在動態投放上嗎？

是的，所有新發布者（視頻雲和一次性發布者）都將配置動態交付。所有現有帳戶都已遷移到動態交付，並且舊的提取系統已關閉。

我的一些HLS格式轉換（根據CMS API）的比特率高於攝取配置文件中指定的比特率-為什麼？

攝取配置文件中指定的是視頻比特率。CMS API報告的是視頻比特率+音頻比特率+動態交付處理的10％開銷。因此，除非源視頻的比特率低於該值，否則CMS API返回的比特率將始終高於攝取配置文件中指定的比特率。
